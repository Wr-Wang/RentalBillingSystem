# 住宅房屋租赁收租结算系统 — 需求文档

> 详细设计请参见[详细设计文档](./住宅房屋租赁收租结算系统-详细设计.md)。

---

## 一、项目核心目标

构建一套专业的 **住宅房屋租赁收租结算系统**，以 BackgroundService + Cronos 为调度引擎，覆盖以下核心业务闭环：

```
基础支撑:
  ├── 组织架构(用户/角色/权限) → 菜单权限控制
  └── 审批工作流(0~N级)       → 合同变更/终止/减免等审批

业务闭环:
房源管理 → 合同签订 → 租金应收 → 多渠道收款 → 账务处理 → 催缴管理 → 报表统计
                                            ↑
                                      Quartz 定时调度
```

**一句话目标**：让公司能管房、管合同、自动算租金、收钱自动记账。

**适用市场**：中国大陆住宅租赁市场  
**会计准则**：CAS 21（企业会计准则第21号—租赁）

## 二、用户使用流程

### 2.1 核心业务流程

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  1.房源   │→│  2.合同   │→│  3.应收   │→│  4.收款   │→│  5.账务   │
│  管理     │  │  管理     │  │  计划     │  │  确认     │  │  处理     │
│  (座楼室) │  │ (签租约)  │  │ (自动生成)│  │ (多渠道)  │  │ (日记账)  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                   │
                                          ┌────────┴────────┐
                                          │  催缴管理        │
                                          │  (逾期→提醒→催缴)│
                                          └─────────────────┘
```

### 2.2 角色操作流程

**运营人员（日常操作）**：
```
登录 → 房屋管理(查看/导入) → 合同管理(新建/变更/续签) 
     → 租客管理(维护信息) → 催缴管理(查看催缴任务)
```

**财务人员（收租对账）**：
```
登录 → 仪表盘(今日收款/欠费) → 收款确认(审核到账) 
     → 银行对账(导入流水→匹配) → 会计凭证(审核)
     → 报表(收租率/欠费明细)
```

**审批人（各级主管）**：
```
登录 → 待审批列表(合同终止/减免/转租等) → 查看申请详情 
     → 通过/驳回 → 自动流转下一级或完结
```

**系统管理员（基础配置）**：
```
登录 → 组织架构(用户/角色) → 菜单权限分配 → 审批流程配置(级别/审批人)
     → 收费项目配置 → 税率配置 → 支付通道管理 
     → 调度任务管理 → 会计科目管理
```

## 四、第一版（MVP）必须实现的功能

MVP 聚焦核心闭环：**房→合同→租金→收款→简单账务**，暂不涉及复杂会计和高级功能。

| 模块 | 功能 | 说明 |
|---|---|---|
| **房屋管理** | 座/楼/室三级CRUD | 基本的增删改查 |
| | 房间房型管理 | 设置房型（整租/合租分类），替代自由文本 |
| | 定价标准矩阵 | 按房型×楼层级别配置标准租金，房间创建时自动建议 |
| | 房间状态管理 | 空置/已租/维修 |
| **租客管理** | 租客信息CRUD | 姓名、身份证、电话 |
| **合同管理** | 合同CRUD（完整状态机） | Draft→PendingApproval→Active→Expired/Terminated/Renewed |
| | 按月支付 | 月付场景为主 |
| | 合同终止 | 到期终止/提前解约（触发审批流程） |
| | 暂停/恢复 | 合同挂起/恢复（暂停期间不生成应收） |
| **费用配置** | 合同费用绑定 | 合同绑定9种费用项目，配置单价/金额 |
| | 按天分摊 | 月中入住/搬出按天计算 |
| **应收账款** | 自动生成应收（全费用） | 每月25日生成下月全部费用的应收计划（Quartz） |
| | 手动触发生成 | 运营手动触发 |
| | 按费用类型拆分 | 应收账单可展开查看各费用明细 |
| **账单存储** | 按合同生成账单快照 | 应收数据生成的同时，按合同（房间）维度和账期生成账单主记录（DebitNotes），存储生成时刻的快照数据，后续多次预览/导出PDF均以此为准 |
| | 仅限未生成过账单的合同可生成 | 每合同每账期只首次生成新账单；已生成过的合同不再允许重新生成，只能导出已存储的历史数据 |

| | 账单历史可追溯 | 同一合同同一账期多次生成账单保留版本记录（VersionNo），支持回溯任一版本 |
| | 多次预览/导出PDF | 账单生成后不限次数预览和导出PDF，数据来自已存储的账单快照而非实时计算 |
| **收款确认** | 收款登记（多计划分配） | 选择多笔应收→输入总金额→按比例或手动分配到各费用项 |
| | 收款确认 | 财务确认到账，生成 ReceiptAllocations |
| | 部分收款 | 一笔应收分多次收 |
| | 收据冲销 | 财务主管+总监双人复核 |
| | 退款 | 多付/贷项余额原路退回 |
| **押金** | 押金收取/退还 | 合同签订收押金、终止退押金 |
| | 押金抵最后月租 | 可选功能，自动抵扣后余款退还 |
| | 扣款（会签审批） | 运营主管+财务主管双方确认 |
| **催缴** | 自动催缴（短信） | 逾期自动发送提醒 |
| | 手动催缴 | 运营手动发起催缴 |

## 五、暂时不做的功能（后续版本）

| 功能 | 原因 |
|---|---|
| **年付租金** | 月付覆盖80%住宅场景，年付的分摊逻辑略复杂 |
| **转租管理** | 涉及合同变更、押金转移、收益分成，复杂度高 |
| **CAS 21 直线法分摊** | 涉及免租期/递增租金的收入分摊，会计准则复杂 |
| **融资租赁** | 住宅场景极少用到 |
| **押金折现** | CAS 21 可选操作，非住宅必需 |
| **发票管理（税控对接）** | 需对接税控盘/电子发票平台，第三方依赖 |
| **支付宝/微信支付对接** | 需商户资质+API对接，可先做银行转账 |
| **电子签章** | 需对接第三方平台（法大大/e签宝） |
| **维修管理** | 辅助功能，不涉及收租核心 |
| **阶梯计价（水电）** | 阶梯水价/电价各地政策不同，先做统一单价 |
| **合同模板+条款库** | 文档复杂度高，MVP用固定合同格式 |
| **自动对账（银企直连）** | 需银行API对接，各地银行接口不同 |
| **减免审批流** | 工作流引擎 |
| **坏账准备** | 会计精细化操作 |

## 六、调度任务并发约束

### 6.1 任务模板与实例

系统预定义四类任务模板，按公司维度独立创建实例：

| 模板 Code | 显示名 | 默认 Cron | 分类 |
|-----------|-------|-----------|------|
| MonthlyFeeBill | 📅 月度应收生成 | `0 0 8 25 * ?` | 计费 |
| LateFeeCalc | 💰 滞纳金计算 | `0 0 2 * * ?` | 计费 |
| AutoRenew | 🔄 自动续签 | `0 0 0 * * ?` | 合同 |
| Collection | 📢 催缴任务 | `0 0 9 * * ?` | 催缴 |

每个公司可以从模板创建自己的任务实例，也可自定义任务。

### 6.2 任务模板管理

| 功能 | 说明 |
|------|------|
| 查看模板 | 系统预置 4 类模板，不可删除修改 |
| 从模板创建 | 选择模板 → 配置 Cron（可选覆盖）→ 为公司创建实例 |
| 自定义创建 | 不选模板，手动输入名称和 Cron |

### 6.3 公司任务实例管理

### 6.2 执行隔离规则

| 场景 | 策略 |
|------|------|
| 不同公司 + 相同 Job | ✅ 允许并行（CompanyId 天然隔离） |
| 相同公司 + 不同 Job | ⚠️ 串行编排（按业务顺序逐个执行） |
| 相同公司 + 相同 Job | ❌ 互斥（防重复执行） |

### 6.3 执行顺序依赖

```
同公司同一周期内的执行顺序（不可颠倒）：

  AutoRenewJob (00:00)
       ↓ 新合同产生新应收
  MonthlyFeeBillJob (每月25日 08:00)
       ↓ 应收数据是滞纳金的基础
  LateFeeCalcJob (每天 02:00)
       ↓ 应收+滞纳金是催缴的依据
  CollectionJob (每天 09:00)
```

### 6.4 数据一致性要求

| 要求 | 说明 |
|------|------|
| **写-读一致** | MonthlyFeeBillJob 写入期间，CollectionJob 必须读到完整的应收数据（Snapshot 隔离） |
| **幂等性** | 重复执行不产生重复应收、重复滞纳金、重复催缴 |
| **事务完整性** | 每个 Job 在显式事务内执行，中途失败自动回滚 |
| **僵死检测** | Job 执行中崩溃，重启后能检测到未完成的任务 |

### 6.5 并发防护级别

| 级别 | 措施 |
|:----:|------|
| 数据库约束 | 唯一索引防止重复 INSERT（应收、账单明细、执行锁） |
| 乐观锁 | RowVersion + 自动重试（CommitWithRetryAsync） |
| 业务逻辑 | Job 执行锁（ScheduledTaskLogs）+ 幂等检查 + 分批提交 |
