# 审批流重设计 — 需求文档

## 1. 背景

现有审批流转模块（`ApprovalService.cs`）在线上持续出现 `DbUpdateConcurrencyException`，状态为"审批通过"时经常因乐观锁冲突导致操作失败。前后经过 4 次代码级别的修复尝试（RowVersion 重试、导航集合清理、Detach+重载、显式 RefreshConcurrencyAsync），均未能解决问题。

经深入排查，**根因不在 RowVersion 本身**，而在于 `MirrorAuditInterceptor` 在 `SavingChangesAsync` 阶段向不存在的 `_Audit` 审计表写入数据时，SQL Server 报错并破坏当前事务，后续的 UPDATE 在被终止的事务上执行 → 0 行影响 → `DbUpdateConcurrencyException`。

## 2. 问题根因

### 2.1 直接原因

```
SaveChangesAsync()
  → 注入审计字段 (UpdatedBy, UpdatedAt)
  → base.SaveChangesAsync()
    → 开启隐式事务
    → [Step 1] MirrorAuditInterceptor.SavingChangesAsync
        → ExecuteSqlRawAsync("INSERT INTO [ApprovalRequests_Audit] ...")
        → 表不存在 → SqlException 208, severity 16
        → SQL Server 终止事务
        → catch {} 静默吞掉异常                ← 掩盖根因
    → [Step 2] DomainEventDispatcher.SavingChangesAsync
    → [Step 3] EF Core 执行 UPDATE 于已终止事务
        → 0 行受影响 → DbUpdateConcurrencyException ← 表象
    → [Step 4] DomainEventDispatcher.SavedChangesAsync (永远走不到)
```

### 2.2 深层原因

| 问题 | 说明 | 影响范围 |
|------|------|---------|
| 审计表缺失 | `ApprovalRequests_Audit`、`Contracts_Audit` 等 19 张 `{Table}_Audit` 表在数据库中不存在 | **全部**修改 `AuditableEntity` 的操作 |
| `catch {}` 静默吞异常 | `MirrorAuditInterceptor` 的空 catch 块掩盖了 SQL 错误 | 错误无法被发现和定位 |
| RowVersion 双重并发 | `Status` 和 `RowVersion` 同时作为并发令牌，后者在审计写入失败时被错误归咎 | 干扰排查方向 |
| 重试逻辑无法恢复 | 每次重试都走同样被破坏的管道，审计写入每次都失败 | 所有重试全部无效 |

## 3. 需求

### 3.1 功能需求

| # | 需求 | 优先级 |
|---|------|--------|
| R1 | 审批通过（Approve）操作必须稳定可靠，不因后台审计写入失败而阻塞 | P0 |
| R2 | 审批驳回（Reject）操作同 R1 | P0 |
| R3 | 审计表缺失不应影响正常业务操作 | P0 |
| R4 | 仍保留乐观锁保护，防止重复审批 | P0 |
| R5 | 审计写入失败的场景应有日志记录，不可静默掩盖 | P1 |

### 3.2 非功能需求

| # | 需求 | 指标 |
|---|------|------|
| NR1 | 审批操作 99.9% 场景下一次 Commit 成功 | ≥99.9% 首次通过 |
| NR2 | 审计写入失败不影响主事务 | 零影响 |
| NR3 | 代码可维护性 | 重试逻辑不超过 2 层嵌套 |

### 3.3 不在此次范围

- 审批流程本身的业务规则变更（审批级别、流转逻辑等）
- 前端 UI 改动
- 现有 `AuditService.GetStatsAsync` 改造
- 领域事件 Outbox 模式

## 4. 方案对比

| 方案 | 描述 | 工作量 | 风险 |
|------|------|--------|------|
| **A. 补建审计表** | 仅创建缺失的 `_Audit` 表 | 小 | 中（不解决架构脆弱性） |
| **B. 审计拦截器修复 + 简化并发** | 修复 `MirrorAuditInterceptor` + 移除 `RowVersion` 双重并发 | 中 | 中（约定覆盖 IsConcurrencyToken） |
| **C. 重写审批模块** | 存储过程或消息队列 Outbox | 大 | 高 |
| **D（选定）. 写路径原始 SQL 化** | 审批写操作完全绕过 SaveChanges 管道，使用原始 SQL + 显式事务 | 中 | **低** |

### 选定方案：D — 写路径原始 SQL 化

核心思路：EF Core 做读，原始 SQL 做写，二者通过显式事务连接，互不干扰。

**为什么方案 B 不可行**：
- EF Core 的 `rowversion` 类型约定会在模型构建后自动覆盖 `.IsConcurrencyToken(false)`
- `SqlServerRetryExecutionStrategy` 在瞬态失败重试时，实体的并发令牌原始值已过期
- `MirrorAuditInterceptor` 的 `ExecuteSqlRawAsync` 运行在 SaveChanges 管道内，任何 SQL 错误破坏事务
- 上述三个问题叠加导致无法在 SaveChanges 管道内彻底解决

## 5. 验收标准

1. `POST /api/approvals/{id}/approve` 正常返回 200，状态变更为 Approved
2. 连续 100 次审批操作无 `DbUpdateConcurrencyException`
3. 并发场景下（10 请求同时审批同一 ID），仅 1 个成功、9 个失败并返回 "已被处理"
4. Import 类型的审批通过后，`ImportBatch.Status` 正确变更为 Approved
