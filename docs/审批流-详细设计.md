# 审批流重设计 — 详细设计

## 1. 背景

### 1.1 问题

`ApprovalService.ApproveAsync` 持续报 `DbUpdateConcurrencyException`，前后经过 5 次迭代修复均失败。

### 1.2 5 次迭代失败记录

| 迭代 | 修复 | 失败原因 |
|------|------|---------|
| 1 | Retry + RowVersion 重试 | 审计表缺失 → MirrorAuditInterceptor 破坏事务 |
| 2 | 清除导航集合（_records） | EF Core 检测到关系被切断 → InvalidOperationException |
| 3 | DetachEntity + 重加载 | 事务仍被破坏，重试无效 |
| 4 | RefreshConcurrencyAsync 显式设 RowVersion OriginalValue | SetValues 可能跳过 rowversion 列 |
| **5 （方案B）** | MirrorAuditInterceptor IF EXISTS + RowVersion.IsConcurrencyToken(false) | **RowVersion 约定仍覆盖配置** + RSES 重试时令牌过期 |

### 1.3 终极根因

**EF Core 的 SaveChanges 管线和本代码库的拦截器配置存在根本性不可调和的问题：**

```
SaveChangesAsync()
  → 审计字段注入 (SetUpdated)
  → MirrorAuditInterceptor.SavingChangesAsync
  │   └─ ExecuteSqlRawAsync                             ← 若审计表缺失，severity 16 破坏事务
  → DomainEventDispatcher.SavingChangesAsync            ← 捕获领域事件
  → SqlServerRetryExecutionStrategy                     ← 瞬态失败重试时并发令牌已过期
  → EF Core UPDATE + INSERT                             ← @@ROWCOUNT = 0 → DbUpdateConcurrencyException
```

四个不可控组件串联在同一管线中，任何一点出错都导致全链路失败。修复单个环节（如 IF EXISTS）仍不能解决其他环节的交互问题。

---

## 2. 总体方案

### 2.1 核心思想

**写操作（Approve/Reject/Cancel）完全绕过 EF Core 的 SaveChanges 管道，使用原始 SQL + 显式事务。**

```
┌─────────────────────────────────────────────────────────────┐
│                        ApprovalService                       │
│                                                             │
│  ┌────────────────┐   ┌────────────────────────────┐        │
│  │  读路径 (EF Core) │   │  写路径 (原始 SQL + 事务)   │        │
│  │  GetByIdAsync   │   │  ApproveAsync              │        │
│  │  GetPendingAsync│   │  RejectAsync               │        │
│  │  MapToDtoAsync  │   │  CancelAsync               │        │
│  └────────────────┘   └────────────────────────────┘        │
│                                │                             │
│                                ▼                             │
│                    ┌──────────────────────┐                  │
│                    │  UnitOfWork           │                  │
│                    │  ├─ ExecuteSqlRawAsync│                  │
│                    │  ├─ BeginTransaction   │                  │
│                    │  ├─ ReloadAsync        │                  │
│                    │  └─ (不调用 CommitAsync)│                  │
│                    └──────────────────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 为什么能彻底解决问题

| 之前的故障点 | 新方案 |
|------------|--------|
| MirrorAuditInterceptor 在 SavingChanges 中破坏事务 | **ExecuteSqlRawAsync 不会触发 SaveChangesInterceptor** |
| DomainEventDispatcher 在 SavedChanges 中分发事件 | **手动 dispatch（独立 DI Scope），与 SaveChanges 解耦** |
| RowVersion 约定覆盖 .IsConcurrencyToken(false) | **原始 SQL 完全控制 WHERE 子句，无 EF 并发令牌介入** |
| SqlServerRetryExecutionStrategy 重试时令牌过期 | **没有 SaveChanges，没有 ExecutionStrategy** |
| 导航集合污染（_records 累积） | **没有 EF 跟踪，只有原始 INSERT 语句** |
| 领域事件重复分发 | **手动 dispatch 一条，精确无误** |

## 3. 架构

### 3.1 一次 Approve 的完整路径

```
POST /api/approvals/{id}/approve
  │
  ├─ ① ApprovalController.Approve(id, body)
  │
  ├─ ② ApprovalService.ApproveAsync(id, comment)
  │
  ├─ ③ [读] _uow.ApprovalRequests.GetByIdAsync(id)
  │     └─ EF FindAsync → 返回跟踪实体
  │
  ├─ ④ 验证 entity.Status == "Pending" — 否则抛异常
  │
  ├─ ⑤ [写] 原始 SQL + 显式事务
  │     ├─ _uow.BeginTransactionAsync()
  │     ├─ _uow.ExecuteSqlRawAsync(
  │     │    "UPDATE ApprovalRequests
  │     │     SET Status = 'Approved', UpdatedBy=@p0, UpdatedAt=@p1
  │     │     WHERE Id = @p2 AND Status = 'Pending'",
  │     │    params)
  │     │    └─ @@ROWCOUNT == 0 → 抛 InvalidOperationException
  │     ├─ _uow.ExecuteSqlRawAsync(
  │     │    "INSERT INTO ApprovalRecords (...) VALUES (...)",
  │     │    params)
  │     └─ tx.CommitAsync()
  │
  ├─ ⑥ [事件] 终审时手动 dispatch ApprovalCompletedEvent
  │     └─ using var scope = _serviceProvider.CreateScope()
  │         var handler = scope.ServiceProvider
  │             .GetRequiredService<IEventHandler<ApprovalCompletedEvent>>()
  │         handler.HandleAsync(event, ct)
  │         └─ 独立 DI Scope → handler 内的操作有独立事务
  │
  └─ ⑦ [读] _uow.ReloadAsync(entity)
        └─ MapToDtoAsync(entity) → 返回 DTO
```

### 3.2 事务边界

```
┌─────────────────────────────────────────────┐
│  BeginTransactionAsync()                     │
│                                             │
│  UPDATE ApprovalRequests                     │  ← 原子：同时成功或失败
│  INSERT INTO ApprovalRecords                 │
│                                             │
│  CommitAsync() / RollbackAsync()            │
└─────────────────────────────────────────────┘

领域事件 dispatch 在 COMMIT 之后，有独立的事务边界。
```

## 4. 详细修改

### 4.1 IUnitOfWork — 新增 ExecuteSqlRawAsync

新增接口方法：
```csharp
/// <summary>执行原始 SQL 命令（绕过 SaveChanges 管道，不触发拦截器）</summary>
Task<int> ExecuteSqlRawAsync(string sql, IEnumerable<object> parameters, CancellationToken ct = default);
```

### 4.2 UnitOfWork — 实现 ExecuteSqlRawAsync

**文件**：`src/RBS.Infrastructure/Data/UnitOfWork/UnitOfWork.cs`

实现：
```csharp
public async Task<int> ExecuteSqlRawAsync(string sql, IEnumerable<object> parameters, CancellationToken ct = default)
{
    return await _context.Database.ExecuteSqlRawAsync(sql, parameters, ct);
}
```

### 4.3 ApprovalService — 写路径全部改为原始 SQL

**文件**：`src/RBS.Application/Services/Approval/ApprovalService.cs`

| 改前 | 改后 |
|------|------|
| 注入 `IUnitOfWork` + `IApprovalDomainService` | 注入 `IUnitOfWork` + `IServiceProvider` |
| `ApproveAsync`：EF Core SaveChanges + retry | **原始 SQL + 显式事务** |
| `RejectAsync`：同上 | **原始 SQL + 显式事务** |
| `CancelAsync`：EF Core SaveChanges | **原始 SQL + 显式事务** |
| `SubmitAsync`：不变（走 EF Core） | 无改动 |
| 领域事件由 DomainEventDispatcher 自动分发 | **手动 dispatch（独立 DI Scope）** |

#### ApproveAsync 核心代码

```csharp
public async Task<ApprovalRequestDto> ApproveAsync(Guid id, string? comment, CancellationToken ct)
{
    // [读] EF Core 加载验证
    var entity = await _uow.ApprovalRequests.GetByIdAsync(id, ct)
        ?? throw new KeyNotFoundException("审批请求不存在");
    if (entity.Status != "Pending")
        throw new InvalidOperationException("该审批已处理，请刷新后重试");

    var now = ChinaTime.Now;
    var userId = _currentUserService.UserId;
    var isFinalLevel = entity.CurrentLevel >= entity.MaxLevel;

    // [写] 原始 SQL + 显式事务
    using var tx = await _uow.BeginTransactionAsync(ct);
    try
    {
        // 终审设 Status='Approved'，非终审进 CurrentLevel+1
        var updateSql = isFinalLevel
            ? "UPDATE ApprovalRequests SET Status='Approved', UpdatedBy=@p0, UpdatedAt=@p1 WHERE Id=@p2 AND Status='Pending'"
            : "UPDATE ApprovalRequests SET CurrentLevel=CurrentLevel+1, UpdatedBy=@p0, UpdatedAt=@p1 WHERE Id=@p2 AND Status='Pending'";

        var rows = await _uow.ExecuteSqlRawAsync(updateSql,
            new object[] { userId, now, id }, ct);
        if (rows == 0)
            throw new InvalidOperationException("该审批已被其他人处理，请刷新后查看");

        // 插入审批记录
        var recordId = Guid.NewGuid();
        await _uow.ExecuteSqlRawAsync(
            "INSERT INTO ApprovalRecords (...) VALUES (...)",
            new object[] { recordId, id, entity.CurrentLevel, userId, "Approved", comment ?? "", userId, now }, ct);

        await tx.CommitAsync(ct);
    }
    catch (InvalidOperationException) { throw; }
    catch
    {
        await tx.RollbackAsync(ct);
        throw;
    }

    // [事件] 终审时手动 dispatch
    if (isFinalLevel)
    {
        using var scope = _serviceProvider.CreateScope();
        var handler = scope.ServiceProvider
            .GetRequiredService<IEventHandler<ApprovalCompletedEvent>>();
        await handler.HandleAsync(
            new ApprovalCompletedEvent(id, entity.TargetEntityId, entity.TargetEntityType, "Approved"), ct);
    }

    // [读] 重新加载返回 DTO
    await _uow.ReloadAsync(entity, ct);
    return await MapToDtoAsync(entity, ct);
}
```

#### RejectAsync 核心逻辑

同 ApproveAsync，区别：
- `UPDATE SET Status = 'Rejected'`（驳回始终为终审）
- `INSERT Action = 'Rejected'`
- 始终 dispatch `ApprovalCompletedEvent`（驳回即终审）

#### CancelAsync 核心逻辑

- 使用 `GetByIdWithRecordsAsync` 加载 Records（需校验提交人）
- `UPDATE SET Status = 'Cancelled'`
- `INSERT Action = 'Cancelled'`
- 若 `TargetEntityType == "Import"`，同时 `UPDATE ImportBatches SET Status = 'Cancelled'`
- 不 dispatch 领域事件（Cancel 不触发 ApprovalCompletedEvent）

### 4.4 MirrorAuditInterceptor — 保留 IF EXISTS 安全屏障

**文件**：`src/RBS.Infrastructure/Data/Interceptors/MirrorAuditInterceptor.cs`

写路径绕过 SaveChanges 后，MirrorAuditInterceptor 不再影响审批流。保留当前 IF EXISTS 修复作为安全屏障，防止其他模块的 SaveChanges 调用被其破坏。

### 4.5 ApprovalRequestConfiguration — 保留配置

**文件**：`src/RBS.Infrastructure/Data/Configurations/Approval/ApprovalRequestConfiguration.cs`

保留 `.IsConcurrencyToken(false)`。虽然写路径已不使用 EF Core 并发令牌，但 EF Core 读取实体时仍由该配置影响行为，保持配置一致性。

### 4.6 AppDbContextFactory — 保留（新增文件）

**文件**：`src/RBS.Infrastructure/Data/AppDbContextFactory.cs`

保留，供 `dotnet ef` 工具在设计时创建 DbContext。

### 4.7 数据库迁移 — 19 张审计表

**文件**：`src/RBS.Infrastructure/Data/Migrations/20260630131731_AddMissingAuditTables.cs`

19 张缺失的 `_Audit` 表已通过迁移创建：`ApprovalRequests_Audit`、`Contracts_Audit` 等。

## 5. Bug 链路对比

### 修复前

```
SaveChangesAsync()
  MirrorAuditInterceptor →
    ExecuteSqlRawAsync("INSERT INTO [ApprovalRequests_Audit] ...")
    表不存在 → SqlException 208 → 事务被终止
    catch {} 静默吞异常
  DomainEventDispatcher → 捕获领域事件
  EF Core UPDATE → 在已终止事务上执行
  @@ROWCOUNT = 0 → DbUpdateConcurrencyException  ← 用户看到的异常
```

### 修复后

```
ApprovalService.ApproveAsync()
  [读] EF Core GetByIdAsync ← 只读，不触发任何拦截器
  [写] BeginTransactionAsync + ExecuteSqlRawAsync
       UPDATE ... WHERE Id=@id AND Status='Pending'  ← 原始 SQL
       INSERT INTO ApprovalRecords ...                 ← 原始 SQL
       COMMIT
       ← 不触发 MirrorAuditInterceptor
       ← 不触发 DomainEventDispatcher
       ← 不触发 SqlServerRetryExecutionStrategy
  [事件] 手动 dispatch ApprovalCompletedEvent（独立 DI Scope）
  [读] ReloadAsync → MapToDtoAsync
```

## 8. 附录：5 次迭代根因时间线

```
迭代 1: 怀疑 RowVersion 缓存 → 加 retry         ❌ 事务已被破坏，重试无用
迭代 2: 怀疑导航集合污染 → ClearPendingRecords ❌ EF 报关系被切断（NoAction 无级联删除）
迭代 3: 怀疑跟踪实体未刷新 → Detach+重加载      ❌ 事务仍被破坏
迭代 4: 怀疑 OriginalValue 设置 → RefreshConcurrency ❌ 同上
迭代 5: 怀疑 MirrorAuditInterceptor → IF EXISTS + IsConcurrencyToken(false)
     ✅ 审计写入不再破坏事务
     ❌ 但 RowVersion 约定覆盖 IsConcurrencyToken(false)
     ❌ RSES 重试时 Status 原始值已过期 → 仍 DbUpdateConcurrencyException

最终方案:
     ✅ 彻底绕过 SaveChanges 管道
     ✅ 原始 SQL + 显式事务
     ✅ 手动 dispatch 领域事件
```
